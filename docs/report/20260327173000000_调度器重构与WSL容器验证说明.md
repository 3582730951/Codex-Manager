# 调度器重构与 WSL 容器验证说明

## 目标

本次改造只动 service、网关和发布脚本，不改前端 UI，重点解决以下问题：

- 单账号固定并发上限导致的热点账号打穿或吞吐浪费
- 多账号在不同配额、不同健康度下无法动态负载均衡
- 下游请求在网关侧因候选跳过、冷却和网络抖动造成的额外等待
- fork 后续同步上游主仓时，调度逻辑散落导致的大面积冲突
- 发布前缺少 `WSL -> Ubuntu -> Docker` 的强制验证闭环

## Fork-safe 设计

为了降低后续合并上游主仓的冲突面，fork 专属逻辑尽量集中在新增 seam，而不是把复杂逻辑散改到上游热点文件中。

新增主 seam：

- `crates/service/src/gateway/routing/scheduler.rs`
- `scripts/tests/docker/*.sh`
- `scripts/tests/wsl_verify_and_promote.ps1`

上游热点文件只做薄接线：

- `crates/service/src/gateway/mod.rs`
- `crates/service/src/gateway/upstream/proxy*.rs`
- `crates/service/src/gateway/observability/metrics.rs`
- `crates/service/src/gateway/routing/cooldown.rs`

这样后续合并上游时，主要关注接线点是否有签名变化；fork 的调度策略、运行时反馈和测试闭环仍可独立维护。

## 调度器架构

### 1. 运行时状态

`scheduler.rs` 为每个账号维护轻量运行时状态：

- 最近分配时间
- 延迟 EWMA
- 成功率 EWMA
- 网络错误惩罚
- 限流惩罚
- 流中断惩罚

这些状态全在内存中维护，不新增重型持久化表，避免增加 SQLite 写放大和同步成本。

### 2. 动态 permit

静态环境变量 `CODEXMANAGER_ACCOUNT_MAX_INFLIGHT` 仍兼容，但默认值已改为 `0`，表示由动态调度器控制，而不是简单关闭保护后裸奔。

动态 permit 基于以下信号计算：

- 账号计划类型
- 最新 usage snapshot 的剩余配额
- route health
- 实时延迟
- 最近网络/429/流中断反馈

这样单账号不再依赖一个固定上限，同时也不会无限放大并发。

### 3. 动态负载均衡

候选账号在原有会话绑定、手动优先、候选筛选之后，再由调度器按评分重排。评分综合：

- 剩余配额
- 健康度
- 延迟
- permit 余量
- 公平性

因此它不再只是“平均轮转请求次数”，而是会把更多负载分配给剩余额度更高、近期更稳、延迟更低的账号。

### 4. 不拒绝下游的等待机制

当候选都因冷却或动态 permit 不可用而被跳过时，执行链路不会立即失败，而是进入短等待窗口，等账号状态更新后再重试候选。

这满足两个约束：

- 不拒绝下游
- 不通过固定节流来压请求

稳定性来自内部排队、permit 恢复和健康反馈，而不是主动丢弃请求。

## 延迟与网络波动修复

本次先在现有架构下消除最明显的等待放大：

- 候选执行阶段在“全被跳过”时改为等待可用窗口，而不是直接耗尽候选
- 冷却与 inflight 更新会主动通知调度器，减少无效轮询
- 最终响应和终止响应都会回写调度反馈，网络错误、流失败、429 会快速影响后续选路

这能降低网络毛刺时持续命中坏账号的概率，也能减少请求在 gateway 内部因为候选空转造成的尾延迟。

## 资源占用优化思路

本次落地的是低冲突、低风险部分，尽量不破坏现有结构：

- 调度器只增加一个全局 `Mutex + Condvar` 状态域，避免引入更多线程
- 账号健康状态放内存，不做数据库热写
- 容器验证脚本在无本地 cargo/pnpm 时使用 Docker 内工具链，并把 Rust/Node 构建产物留在容器临时层，避免污染宿主工作树
- 发布脚本要求工作树完全干净，避免测试垃圾、未跟踪文件或临时产物被误发到远端

## WSL / Docker 验证闭环

新增脚本：

- `scripts/tests/docker/verify_and_test_stack.sh`
- `scripts/tests/docker/promote_running_stack.sh`
- `scripts/tests/docker/cleanup_test_stack.sh`
- `scripts/tests/wsl_verify_and_promote.ps1`

执行原则：

1. 在 WSL Ubuntu 中运行验证脚本
2. 若 WSL 内没有 `cargo` 或 `pnpm`，自动退回 Docker 工具链容器执行测试和构建
3. 启动临时 compose project 验证 service/web 健康
4. 如配置了 `CODEX_API_KEY` 或 `OPENAI_API_KEY`，继续跑网关回归脚本
5. 只有在验证成功后，才允许进入 push + 正式容器更新
6. 测试结束后清理测试容器、网络、镜像、卷和 override 文件

## 现阶段边界

本次没有改前端 UI，也没有把整个 HTTP 层完全迁到新的异步服务框架。更深一层的 runtime 收口仍可继续做，但需要更大范围地重构入口、流式桥接和后台任务模型。

当前落地结果更适合作为 fork 的第一阶段：

- 先把调度决策从固定阈值升级成动态策略
- 先把验证闭环补齐
- 先把后续合并上游的冲突面压缩到少数 seam
