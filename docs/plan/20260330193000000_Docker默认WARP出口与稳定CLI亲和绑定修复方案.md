# Docker 默认 WARP 出口、稳定 CLI 亲和绑定与 SSE 断流修复方案

## Summary

本文档统一定义 3 个改造方向：

1. WARP 为可选配置，默认不启用；启用后也只用于“下游 CLI 已绑定 ChatGPT 账号后，该账号向上游实时转发请求”的链路。
2. CLI 账号绑定从“`session_id` 优先”改成“稳定 CLI 键优先”，修复当前无报错也换号的问题。
3. 修复 `/v1/responses` 的 SSE bridge 终态判定与收尾同步问题，消除错误的 `502 stream_interrupted`。

当前结论已经确认：

- 当前“无错误切号”主要不是同一 binding 被偷偷改号，而是 `derive_affinity_key_from_parts()` 仍然 `session_id` 优先。
- 当前 `client_bindings` 在同账号成功请求时也会递增 `binding_version`，这会放大“像是切号”的错觉，语义也不对。
- 当前仅发现 2 个 `502` 样本，均为单账号、单尝试、上游先 `200` 后流式未见 terminal event，不是调度切号导致。
- 当前 `502` 触发点在 `response_finalize.rs`：`client_is_stream && !bridge.stream_terminal_seen`。
- 当前 bridge 存在 collector 快照竞态风险，日志已经出现 `terminal_seen=false && terminal_error=None`，这说明最终判定时序不可靠。

## 1. WARP 作用域与默认行为

WARP 是可选项，默认不启用。

只有显式配置 `CODEXMANAGER_GATEWAY_ACCOUNT_PROXY_URL` 后，才会启用该专用出口。

WARP 只作用于以下路径：

- 下游 CLI 已绑定账号
- 账号类型为 ChatGPT 账号
- 请求属于实时上游转发 `/v1/responses -> chatgpt.com/backend-api/codex/responses`

WARP 不覆盖以下路径：

- usage refresh
- token refresh
- OAuth code exchange
- aggregate API
- Azure OpenAI
- API key 直连路径
- 其他后台任务 HTTP

如需启用，可在 Docker 场景新增 `warp-egress` sidecar，并配置专用环境变量：

- `CODEXMANAGER_GATEWAY_ACCOUNT_PROXY_URL=socks5://warp-egress:1080`

关键约束：

- 不复用全局 `CODEXMANAGER_UPSTREAM_PROXY_URL`
- 只在 ChatGPT 账号实时转发 client 构建链路里消费该变量
- WARP sidecar 不可用时，同账号先做一次 direct fallback
- 只有 direct fallback 也失败时，才进入现有账号失败逻辑
- 不允许因为 WARP sidecar 故障而直接切到别的账号

## 2. 稳定 CLI 亲和绑定修复

当前问题：

- 现有 affinity key 优先级是 `session_id > conversation_id > x-openai-subagent > x-client-request-id`
- 这会导致同一个下游 CLI 只要换了 `session_id`，系统就创建新的 binding，看起来像“没报错也换账号”

修复后默认优先级固定为：

- `x-codex-cli-affinity-id > conversation_id > session_id > x-openai-subagent > x-client-request-id`

新增规则：

- 新增下游专用头 `x-codex-cli-affinity-id`
- 该头只用于本地入站亲和识别，绝不透传上游
- 未显式提供 CLI 键时，默认退化为 `conversation_id` 级稳定，不再让 `session_id` 抢第一优先级

binding 提交逻辑同时调整：

- 同账号成功请求只执行 `touch_client_binding`
- 不再每次成功都重写 `client_bindings`
- 不再每次成功都递增 `binding_version`
- 只有以下情况才递增：
  - 首次绑定
  - 实际迁移到新账号
  - `primary_scope_id` 发生变化
- 只有真实迁移才写 `last_switch_reason`

日志与可观测性补充：

- `affinity_source`
- `affinity_key_fp`
- `binding_action=reuse|bind|migrate`
- `existing_account_id`
- `chosen_account_id`
- `switch_reason`

这样可以直接区分：

- 旧 binding 复用
- 新键首次绑定
- 真正的切号迁移

## 3. SSE 502 `stream_interrupted` 修复

当前问题不是“候选切号”，而是 bridge 终态判定不可靠。

已确认现象：

- 上游 `ATTEMPT_RESULT status=200`
- 后续 `BRIDGE_RESULT terminal_seen=false`
- `last_sse_event` 停在 `response.created`、`response.output_text.delta` 或 `response.function_call_arguments.delta`
- 最终被记成 `502 stream_interrupted`

需要同时修 2 类问题：

### 3.1 终态分类

桥接层要显式区分以下状态：

- `terminal_ok`
- `terminal_err`
- `eof_without_terminal`
- `reader_error`

规则固定：

- `response.completed` / `response.done` / `[DONE]` 才算正常完成
- `response.failed` / `*.error` / `*.incomplete` 算上游终态失败
- 收到 EOF 但没见 terminal，记为 `eof_without_terminal`
- reader 本身报错，记为 `reader_error`

### 3.2 收尾同步

当前 `request.respond(response)` 返回后，bridge 就立刻克隆 collector 作为最终结果，这里存在竞态。

修复要求：

- 为 `Passthrough/OpenAI` reader 增加 completion barrier
- `respond_with_upstream()` 必须等待 reader 进入最终 completion state 后再给出 bridge 结果
- 禁止再出现 `terminal_seen=false && terminal_error=None` 这种无结论状态
- 若等待超时，要明确记录 `completion_wait_timeout`

账号惩罚逻辑拆分：

- `reader_error` 或明确连接中断，才按网络故障进入 cooldown
- `eof_without_terminal` 记 `stream_interrupted`
- `eof_without_terminal` 不再直接把账号标记 unavailable
- `eof_without_terminal` 也不直接按网络故障降权

## 4. 关键接口与代码边界

本次改造只收敛在 3 个区域：

- `gateway/affinity/*`
- `gateway/observability/http_bridge/*`
- Docker compose / WARP sidecar 配置层

不扩大到以下区域：

- 现有评分公式主体
- mid-stream 切号
- OAuth 流程
- usage/account background pipeline 主逻辑

新增接口与配置：

- 入站头：`x-codex-cli-affinity-id`
- Docker env：`CODEXMANAGER_GATEWAY_ACCOUNT_PROXY_URL`
- trace 字段：`affinity_source`、`affinity_key_fp`、`binding_action`、`completion_state`

## 5. 容器化验证方案

只允许使用：

- WSL Ubuntu
- Docker 测试容器
- fake token
- mock upstream

禁止：

- 真实 OpenAI key
- 真实 ChatGPT 账号上游作为验收环境

必测场景：

- `x-codex-cli-affinity-id` 固定、`session_id` 变化，账号不变
- 无 CLI 键、`conversation_id` 固定、`session_id` 变化，账号不变
- 同账号成功请求不再递增 `binding_version`
- `response.completed` 正常结束，不得误报 502
- `response.created` 后断流，必须报 `stream_interrupted`
- `response.output_text.delta` 后断流，必须报 `stream_interrupted`
- `response.function_call_arguments.delta` 后断流，必须报 `stream_interrupted`
- WARP sidecar 不可达时，同账号 direct fallback 成功
- WARP sidecar 不可达时，不得仅因代理故障切到别的账号
